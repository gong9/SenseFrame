# SenseFrame 小宫产品控制层规划

> [!summary]
> 小宫不应该只是一次性的“审片按钮”，也不应该只是右下角的单张评价。长期来看，小宫应该是 SenseFrame 的产品灵魂：它理解整个产品状态，能通过工具调用控制产品功能，能和摄影师对话完成查找、筛选、审片、解释、重排和导出准备，但始终不越过用户确认去删除原片或覆盖人工审美。

## 一句话定位

小宫是 SenseFrame 里的摄影审片副驾和产品控制层。

它不是通用聊天机器人，而是一个懂摄影选片、懂当前批次、懂产品功能、能调用 SenseFrame 内部工具的工作台智能体。

用户可以像对助理说话一样提出目标：

```text
给我找出你认为最好看的图片。
把这组里面闭眼误判的找出来。
这批照片先帮我整理成精选、待定和明显问题。
找出适合做封面的 20 张。
把近重复里每组最值得看的列出来。
```

小宫理解目标后，不是只返回文字，而是操作产品：

- 切换或生成照片列表。
- 高亮推荐照片。
- 更新 AI 分组视图。
- 解释筛选理由。
- 标记需要人工复核的冲突点。
- 准备导出清单。
- 在用户确认后应用 pick / maybe / reject 等人工决策。

## 产品原则

### 1. 小宫是产品能力的统一入口

SenseFrame 的功能应该逐步工具化、命令化、可被小宫调用。

产品不只是 UI 按钮集合，而是一组内部能力：

- 导入批次。
- 读取批次状态。
- 分析照片。
- 搜索照片。
- 获取 AI 分组。
- 获取相似组。
- 视觉审片。
- 组内比较。
- 保存大脑分组。
- 记录人工反馈。
- 应用人工标记。
- 导出已选。

UI 是人手动操作这些能力的入口，小宫是语言和目标驱动的入口。

### 2. 小宫有完整产品上下文

小宫回答问题和执行任务时，必须读取真实产品状态，而不是凭空生成建议。

上下文包括：

- 当前 batch。
- 当前照片。
- 当前左侧 AI 分组。
- 当前筛选模式。
- 小模型分析结果。
- 大脑审片结果。
- 人工 pick / reject / maybe / rating。
- 近重复组和相似连拍。
- 语义分析和搜索结果。
- 历史反馈和用户偏好。
- 可执行工具和权限边界。

### 3. 小宫能操作产品，但不能替代摄影师最终审美

小宫可以自动生成自己的 AI 分组结果，也可以给出操作建议。

但这些动作必须有边界：

- 可以自动写入 `brain` 结果表。
- 可以自动更新“小宫视图”和 AI 分组展示。
- 可以自动生成候选清单。
- 可以自动记录推理日志。
- 不自动删除、移动、重命名原片。
- 不静默覆盖人工 pick / reject / maybe。
- 不自动导出最终交付结果。
- 涉及人工 decision、原片、导出、批量应用时必须确认。

### 4. 聊天入口不是新产品，而是工作台控制面板

小宫聊天入口不应该把 SenseFrame 变成一个对话应用。

正确形态是：

- 主图仍然是中心。
- 左侧分组仍然是核心导航。
- 胶片条仍然用于快速审片。
- 右侧 inspector 显示技术信息、大脑评价和聊天控制。
- 聊天是附着在工作台里的控制面板，不是独立页面。

小宫说完之后，产品界面应该跟着变化。

例如用户说：

```text
给我找出你认为最好看的图片。
```

理想结果不是只回复“我找到了 18 张”，而是：

- 左侧出现或切到“小宫精选”临时视图。
- 中间展示第一张推荐图。
- 胶片条列出推荐清单。
- 右侧显示选择策略和当前照片理由。
- 顶部显示本次任务摘要。

## 小宫的能力分层

### 第一层：只读理解能力

小宫先能看懂产品状态。

需要的能力：

- 读取批次列表。
- 读取当前 batch overview。
- 读取当前照片详情。
- 读取 AI 分组和数量。
- 读取照片分析结果。
- 读取相似组和连拍段。
- 读取大脑历史运行结果。
- 读取人工选择和评分。

典型问题：

```text
这批照片现在有多少张需要复核？
为什么精选候选里有这张？
这组近重复里你推荐哪张？
小模型和你判断冲突最多的地方是什么？
```

### 第二层：可视筛选能力

小宫能根据目标生成一个产品内照片列表。

这类任务不一定改数据库，只改变当前工作台视图。

典型任务：

```text
给我找出你认为最好看的图片。
只看适合做封面的。
找出闭眼可能误判的照片。
把有人物表情自然但技术分低的列出来。
找出每组连拍里最值得看的 1 张。
```

输出应该是一个 `SmartView`：

- view id。
- query / task。
- photo ids。
- ranking。
- reason by photo。
- summary。
- created by Xiaogong。
- 是否临时视图。

### 第三层：审片执行能力

小宫能执行批次级审片流程。

这对应现在已有的 `小宫审片`，但后续应该作为聊天也能触发的工具。

典型任务：

```text
帮我重新审一遍这批。
先把这批整理成可交付候选、待定、明显问题。
重点帮我复核闭眼和连拍。
用封面优先策略跑一遍。
```

执行流程：

1. 建立批次上下文。
2. 制定审片计划。
3. 选择需要视觉模型真实看图的照片。
4. 复核风险照片。
5. 比较近重复和相似连拍。
6. 生成统一大脑审片状态。
7. 更新 AI 分组展示。
8. 给出摘要和下一步建议。

### 第四层：产品操作能力

小宫能调用产品内部命令改变工作台状态。

可自动执行的操作：

- 切换当前照片。
- 切换当前分组。
- 创建临时智能视图。
- 排序胶片条。
- 展示对比组。
- 打开 debug overlay。
- 展示大脑日志。
- 记录大脑反馈。

需要确认的操作：

- 批量应用 pick / maybe / reject。
- 修改 rating。
- 导出已选。
- 删除批次。
- 删除原片。
- 覆盖已有人工 decision。

### 第五层：偏好学习能力

小宫应该逐步理解摄影师偏好。

偏好来源：

- 用户采纳了哪些建议。
- 用户拒绝了哪些建议。
- 哪些照片最终被 pick。
- 哪些照片被导出。
- 用户给了几星。
- 用户反复从待定里选出了什么类型。
- 用户口头提出的策略，比如“这批偏纪实，不要只选干净构图”。

偏好不应该一开始就复杂训练模型，可以先做结构化记忆：

```json
{
  "preference": "活动纪实优先保留关键动作，允许轻微模糊",
  "source": "user_instruction",
  "scope": "batch",
  "created_at": "2026-05-03T00:00:00.000Z"
}
```

## 产品工具化设计

### 工具分类

#### Read Tools

只读，不改变产品状态：

- `ListBatches`
- `GetCurrentWorkspaceState`
- `GetBatchOverview`
- `ListPhotos`
- `GetPhotoDetail`
- `GetBucketAssignments`
- `GetSimilarGroups`
- `GetBrainRun`
- `GetReviewHistory`

#### View Tools

改变当前 UI 视图，但不写入审片结果：

- `ShowPhoto`
- `ShowBucket`
- `CreateSmartView`
- `ShowSmartView`
- `ShowCompareGroup`
- `SetFilmstripOrder`
- `ToggleDebugOverlay`

#### Brain Tools

写入小宫自己的审片状态，不改人工 decision：

- `StartBrainReview`
- `ReviewPhotoVisual`
- `CompareGroupVisual`
- `ReviewRiskVisual`
- `SaveBrainAssignments`
- `SaveBrainGroupRanking`
- `RecordBrainEvent`

#### Confirmed Action Tools

必须用户确认：

- `ApplyDecision`
- `BatchApplyDecisions`
- `SetRating`
- `ExportSelected`
- `DeleteBatch`
- `DeleteOriginalFiles`

### 工具调用权限

小宫需要有权限分级，不能所有工具都一样。

| 权限级别 | 可执行动作 | 是否需要确认 |
| --- | --- | --- |
| `read` | 读取批次、照片、分组、日志 | 否 |
| `view` | 切换视图、创建临时列表、排序展示 | 否 |
| `brain_write` | 写入小宫审片结果、组内排名、事件日志 | 否 |
| `decision_write` | 应用 pick / maybe / reject / rating | 是 |
| `export` | 导出已选、生成清单 | 是 |
| `destructive` | 删除批次、删除原片 | 强确认 |

### 工具结果必须可解释

每个工具调用都应该留下事件：

- 小宫为什么调用这个工具。
- 输入范围是什么。
- 影响了哪些照片。
- 是否写入数据库。
- 是否需要用户确认。
- 失败原因是什么。

这不是为了技术日志好看，而是为了让摄影师信任小宫。

## 聊天入口设计

### 入口位置

建议先放在右侧 inspector 的大脑区域，作为“小宫控制台”。

工作台右上角保留 `小宫审片` 快捷按钮。

右侧新增：

- 当前任务状态。
- 输入框。
- 建议指令 chips。
- 最近小宫动作。
- 当前视图解释。

### 默认建议指令

不要放泛泛的“你想做什么”。要给摄影师能直接点的任务：

- 找出最好看的。
- 复核闭眼误判。
- 每组连拍选 1 张。
- 找封面候选。
- 找需要人工判断的。
- 解释当前照片。
- 重新整理这个批次。

### 对话输出形态

小宫回复应分成两部分：

1. **对摄影师说的话**
   简短说明结果和下一步。

2. **产品状态变化**
   照片列表、当前视图、分组、排序、建议动作。

示例：

```text
我先按“成片观感 + 表情自然 + 背景干净 + 组内代表性”找出了 24 张。
其中 7 张我建议你优先看，5 张和小模型判断有冲突。
```

产品同时切换到“小宫精选 24”视图。

## 关键用户故事

### 用户故事 1：找最好看的图片

用户说：

```text
给我找出你认为最好看的图片。
```

小宫应该：

1. 读取当前 batch。
2. 判断是否已有 completed brain run。
3. 如果没有，提示需要先跑快速审片或直接基于小模型生成临时结果。
4. 结合 visual scores、人工 pick、组内代表性、风险 flags。
5. 创建“小宫精选”智能视图。
6. 按推荐优先级排序。
7. 解释选择策略。

结果：

- 中间主图显示排名第一。
- 胶片条显示推荐列表。
- 右侧显示当前照片入选理由。
- 用户可以逐张确认 pick。

### 用户故事 2：闭眼误判复核

用户说：

```text
把你觉得闭眼误判的找出来。
```

小宫应该：

1. 找出小模型标记 `closed_eyes` 或 `eyes_uncertain` 的照片。
2. 优先看高分、有人物、组内代表、有人工 pick 的照片。
3. 判断是真闭眼、笑眼、情绪闭眼、侧脸遮挡还是误检。
4. 创建“闭眼误判复核”视图。
5. 把真正失败闭眼、可保留闭眼、需要人工判断分开。

### 用户故事 3：每组连拍选代表图

用户说：

```text
每组连拍帮我选一张最值得看的。
```

小宫应该：

1. 读取相似连拍和近重复组。
2. 每组筛出候选。
3. 对候选进行视觉比较。
4. 生成 group ranking。
5. 创建“每组代表图”视图。
6. 保留备选和淘汰理由。

### 用户故事 4：批量应用建议

用户说：

```text
把这些都标成保留。
```

小宫不能直接执行。

它应该显示确认：

```text
将 24 张照片标记为保留，已有 3 张是待定，0 张是人工淘汰。是否确认？
```

确认后才调用 `BatchApplyDecisions`。

## 信息架构

### 左侧

保留现有 AI 分组。

新增临时或持久智能视图区域：

- 小宫精选。
- 封面候选。
- 闭眼误判复核。
- 每组代表图。
- 人工冲突。
- 本次任务结果。

### 中间

主图继续作为审片中心。

显示最少必要状态：

- 当前小宫视图名称。
- 当前推荐排名。
- 是否需要人工复核。
- 是否和人工 decision 冲突。

### 右侧

右侧成为“小宫交接区”：

- 小宫聊天输入。
- 当前任务进度。
- 当前照片解释。
- 建议动作。
- 采纳 / 不采纳。
- 查看日志。

### 底部胶片条

胶片条应该能承载小宫生成的排序：

- 推荐排序。
- 组内排名。
- 风险优先级。
- 人工复核优先级。

## 技术架构

### 总体结构

```text
React Workbench
  -> Xiaogong Chat Panel
  -> IPC
  -> Xiaogong Orchestrator
  -> Tool Registry
  -> Product Services
  -> SQLite
  -> Vision Model / Local Models
```

### Orchestrator 职责

`Xiaogong Orchestrator` 是小宫的大脑编排层。

它负责：

- 解析用户意图。
- 读取工作台上下文。
- 选择工具。
- 控制权限。
- 执行多步任务。
- 生成智能视图。
- 写入事件日志。
- 返回 UI 状态变化。

它不应该直接操作数据库细节，而是调用产品 services。

### Tool Registry 职责

所有可被小宫调用的产品功能都注册为工具。

每个工具声明：

- name。
- description。
- input schema。
- output schema。
- permission level。
- dry-run support。
- confirmation requirement。
- handler。

### UI State Patch

小宫每次任务完成后，应该返回一个 UI patch，而不是只返回文字。

示例：

```json
{
  "message": "我找出了 24 张小宫精选。",
  "uiPatch": {
    "mode": "smartView",
    "smartViewId": "view_123",
    "activePhotoId": "photo_001",
    "inspectorTab": "xiaogong"
  }
}
```

前端根据 patch 切换界面。

## MVP 落地路线

### 阶段 1：小宫智能视图

目标：让“小宫找图”先成立。

范围：

- 新增 smart view 数据结构。
- 新增 `CreateSmartView` / `ShowSmartView`。
- 支持“给我找出最好看的图片”。
- 基于现有 `brainReview`、小模型分数、人工 pick 和组内代表性排序。
- 不新增复杂聊天编排，先做单轮命令。

完成标准：

- 用户输入一句话后，产品切到小宫生成的照片列表。
- 每张照片有入选理由。
- 不改人工 decision。

### 阶段 2：小宫聊天入口

目标：让右侧出现可用的小宫控制台。

范围：

- 新增聊天面板。
- 支持 5 到 8 个固定意图。
- 每个意图映射到工具链。
- 展示工具执行过程。
- 返回 UI patch。

固定意图：

- 找最好看的。
- 找封面候选。
- 复核闭眼误判。
- 每组连拍选代表图。
- 找人工冲突。
- 解释当前照片。
- 重新审片。

### 阶段 3：工具注册和权限系统

目标：把产品功能正式工具化。

范围：

- Tool Registry。
- 权限等级。
- confirmation flow。
- tool events。
- dry-run。
- 错误恢复。

### 阶段 4：多步任务编排

目标：小宫能完成真正复杂任务。

范围：

- 任务计划。
- 多工具调用。
- 中间状态展示。
- 可暂停、可继续、可重试。
- 支持策略参数。

### 阶段 5：偏好学习

目标：让小宫越用越像摄影师本人。

范围：

- 记录采纳和拒绝。
- 记录导出结果。
- 生成偏好摘要。
- 在下一批审片时作为上下文注入。

## MVP 优先实现的第一条能力

建议第一条就做：

```text
给我找出你认为最好看的图片。
```

原因：

- 用户价值直观。
- 不涉及破坏性写入。
- 可以复用现有 brainReview 和小模型结果。
- 能自然验证小宫是否真的能控制产品视图。
- 是聊天入口从“回答问题”变成“操作产品”的最小闭环。

最小闭环：

```text
用户输入
  -> 意图识别 best_photos
  -> 读取 batch
  -> 排序照片
  -> 创建 smart view
  -> 切换 UI
  -> 展示理由
```

## 成功标准

小宫产品控制层真正成立，不是看有没有聊天框，而是看：

- 用户一句自然语言能改变工作台状态。
- 小宫生成的是可操作照片列表，不只是文字建议。
- 小宫能解释每张图为什么出现。
- 小宫能调用已有产品能力，而不是绕开产品另做一套。
- 小宫能遵守权限边界。
- 人工审美仍然是最终确认者。
- 所有小宫动作都有日志、可追踪、可复核。

## 设计结论

小宫应该成为 SenseFrame 的产品灵魂：它理解摄影选片任务，也理解 SenseFrame 的所有工具。用户不是在和一个聊天机器人闲聊，而是在把目标交给一个能控制工作台的摄影助理。

短期先做“小宫找图”和智能视图，打通自然语言到产品状态变化的闭环。中期再把审片、分组、对比、导出准备都工具化。长期让小宫基于用户反馈形成摄影偏好，成为真正懂这个摄影师的审片副驾。
